Project Background

Motivation

This project all started at the beginning of the semester when our group first came together, and a team was born. As a team, we all collectively grew up playing and loving video games which was a common interest amongst us. Growing up in an environment surrounded by video games bred the passion for each of our members and as such this was the perfect opportunity to create something that would allow others to also share this passion our group has. As a result, we decided that trying to recreate a text-based game would be a fun and innovative way to show other people what video games are all about. With text-to-game, we are providing a platform for both authors and users to share their innovative and imaginative ideas that would allow them to create their own text-based game. Authors can use our program to create playable chapters and stories in which they can then publish which would allow other users looking to play text-based games to give it a shot. This would provide authors who want to create a game but do not necessarily have the programming skills an opportunity to create their own text-game which was what our group was trying to accomplish.

Aims and Goals

Aim

The aim of text-to-game is to create an interactive playable program which would allow authors to create their own games along with their own chapters and storylines and for it to be playable by other users. We hope to create a simple yet effective program which would provide the bare backbone for authors to get started with their game. As a result, in achieving this aim, there are a few goals that must be accomplished for the aim to become a reality. These goals are as following:

Goals

'
Ability to create chapters An important goal that we must accomplish is giving the authors the ability to create their own chapters. Since the aim of our group is to create a project that allows authors to create their own games, it is vital that authors are given the ability to create their own individual storylines and chapters.
Storage databaseTo store the information and chapters, the program must have a database in which the information can be gathered from. The database would be used to store all the necessary information that the author may choose to add into their text-to-game. The database would hold data such as chapters, characters, maps and many more.
The option to import images The option to import images would also be a goal that our program ideally would have. Importing images would allow the users to visualize the activities or environment that they are in whilst they are playing the game. Therefore, this goal would help bring our program to the next level as images are a vital aspect in games as they help immerse the user into the game and make it feel like they are a character within the game.
Compatible with user device To accomplish our aim of having a playable program, the program must be compatible with the device of the author or the user. Therefore, this goal is about ensuring that our program is compatible or can be run with most popular devices or operating systems out there. To achieve this goal, our program will be created using java which is a very common application that is compatible with almost all devices

Scope

Project Progress

The idea of Text-to-Game began from the initial group meeting on Microsoft Teams between a few members. Once the idea was followed up to other members and agreed to, we began to work towards receiving confirmation and validity of our idea. We pitched the product, providing descriptions while reviewing the purpose and aim of our project. Throughout the project, the original plan of creating a program that allowed others to create a text game remained intact. It was only until much later into the project that we would decide to create a story using the Text-to-Game application as an additional part of our presentation. We also want to present the application as it is with a functioning user interface.

Generally, our group did not experience much of a scope creep since we mainly kept towards completing the Minimal Viable Features (MVFs). However, if we had our time again, we would most likely try to identify different MVFs that are more significant to the project as we did receive verbal feedback that our MVFs seemed a bit more like EVFs.

We had some issues while using Trello which mainly had to do with lack of consistent use from most members. Sometimes the task would not be enough to achieve full marks during the weekly stand-ups so members would have to allocate themselves extra tasks to fulfill the criteria. The first few weeks were fine with all members following the Trello plan, but it eventually saw a slow decline in usage. The plan increasingly became more confusing to follow and resulted in most members exclusively using Trello to occasionally upload work that was set through discussion in team meetings. Perhaps the Trello plan could have been more carefully fleshed out from the very beginning as it seemed to be a bit rushed.
